CLAUDE CODE 環境設計ガイド

成果を決めるのは、
指示ではなく環境である。

Claude Codeを使い始めると、ある壁に当たります。「うまくいく日と、いかない日がある」。その差を生んでいるのはプロンプトの上手さではなく、CLAUDE.md・フォルダ構造・知識の置き場——つまり環境設計です。このガイドは、環境を「育てられる人」になるための実務指針をまとめたものです。

想定している読者
  • Claude Codeを数週間使っていて、CLAUDE.mdが何かは知っている
  • 「もっと安定して良い結果を出したい」「毎回同じ説明をするのに疲れた」と感じている
  • フォルダやルールをどう整えるべきか、体系的な指針が欲しい
01

なぜ「環境設計」がすべてなのか

同じモデル・同じ指示でも、人によって結果がまるで違う。その理由から始めます。

Claude Codeは、セッションが変わるたびに記憶ゼロの状態で「出社」してきます。前の会話も、あなたの好みも、プロジェクトの事情も覚えていません。毎回が初出勤です。

ここで2つの職場を想像してください。マニュアルも整理された書庫もなく、毎朝口頭で全部説明する職場。もう一方は、机の上に簡潔な指示書が置いてあり、書庫は整理され、よくある業務は手順書になっている職場。同じ優秀な新人でも、成果は環境で決まります。Claude Codeもまったく同じです。

プロンプト（その場の指示）は1回きりしか効きません。環境は毎回、自動で効きます。だから上達の投資先は、指示の言い回しではなく環境なのです。

環境をつくる5つの要素

このガイドで扱うのは次の5つ。それぞれが章に対応しています。

2

CLAUDE.md

毎回必ず読まれる指示書。環境の心臓部。

3

フォルダ構造

AIが世界を理解するための地図。

4

知識の置き場

事実・状態・好みをどこに記録するか。

5

定型業務をコマンドに固めて品質を安定させる。

「守ってほしい」を「必ず守られる」に変える安全装置。

たとえるなら

環境設計とは「AIという新入社員のための職場づくり」です。指示書（CLAUDE.md）、整理された書庫（フォルダ）、業務台帳（ナレッジ）、手順書（コマンド）、決裁権限規程（パーミッション）。人間の組織で効くものは、AIにもそのまま効きます。

この章の要点
  • Claudeは毎セッション記憶ゼロで出社する。だから環境がすべて
  • プロンプトは1回きり、環境は毎回効く。投資対効果が違う
  • 環境の5要素＝CLAUDE.md／フォルダ／知識の置き場／部品化／権限
02

CLAUDE.md設計の原則

環境の心臓部。「何を書くか」より先に「何を書かないか」を知ると、一気に安定します。

まず、3階層の役割分担

CLAUDE.mdは複数の場所に置けて、上から順に重ねて読み込まれます。それぞれ役割が違います。

場所効く範囲書くべきもの
~/.claude/CLAUDE.mdすべてのプロジェクトあなた個人の普遍ルール（言語・文体・確認の取り方・禁止事項）
プロジェクト/CLAUDE.mdそのプロジェクト全体事業・プロジェクトの前提、フォルダの意味、保存先ルール
サブフォルダ/CLAUDE.mdそのフォルダでの作業その案件・部門だけの固有ルール

判断基準はシンプルで、「このルールはどの範囲で守ってほしいか」。全プロジェクト共通なら個人用へ、その現場だけならプロジェクトへ。置き場所を間違えると、関係ないプロジェクトにまでルールが漏れたり、必要な場所で効かなかったりします。

4つの原則

原則 1

「指示」だけを書く。「状態」を書かない

「〜PJは現在デザイン修正中」のような進捗・状態を書くと、終わったあとも毎回読み込まれて、古い情報を最新として扱う事故の温床になります。状態は専用の進捗メモへ（第4章）。CLAUDE.mdは賞味期限のないルールだけ。

原則 2

短く、具体的に、箇条書きで

「きれいに書いて」ではなく「見出しは2階層まで・語尾は敬体」。読んだClaudeが迷わない粒度まで具体化します。曖昧なルールは、無いのと同じです。

原則 3

「するな」より「こうして」

禁止だけ書くと、Claudeは代わりの正解を推測で選びます。「Excelを使うな」ではなく「表計算はGoogle Sheetsで作る（Excel禁止）」。正しい行き先までセットで示します。

原則 4

肥大化したら「分離」する

何でも足していくと数百行に膨らみ、大事なルールが埋もれます。目安は200行。超えてきたら詳細を別ファイル（rules/ など）に移し、本体には絶対ルールと@import（参照）だけを残します。

悪い例と良い例

ありがちなCLAUDE.md（伸び悩む）
# メモ
・このプロジェクトはLPのリニューアル中。
  いまはヒーローセクションを直している（6/10）
・ちゃんとした文章で書くこと
・前に頼んだ資料のフォーマット、
  ああいう感じでお願いします
・エラーが出たときの対処法メモ：（40行続く…）

進捗（すぐ古くなる）・曖昧な指示（「ちゃんとした」）・会話の文脈依存（「ああいう感じ」）・長大なメモが混在。毎回読み込まれるのに、毎回役に立たない。

設計されたCLAUDE.md（安定する）
# このプロジェクト
飲食店向けLP制作。読者は店舗オーナー。

# ルール
- 文体：敬体。1文は60字以内
- 表はGoogle Sheets（Excel禁止）
- 生成物は output/ に保存
- 削除の前は必ず確認を取る

# 詳細ルール（必要時に参照）
@rules/writing-style.md
@rules/design-tokens.md

前提→ルール→参照の3ブロック。すべて賞味期限のない指示。詳細は分離してあり、本体は1分で読み切れる。

育て方のコツ：CLAUDE.mdは最初に完璧を目指すものではなく、運用しながら1行ずつ足すものです。Claudeが期待と違う動きをしたら、その場で「これはルール化すべきか？」と考える。この習慣が数週間で大きな差になります（第4章の育成ループ）。
この章の要点
  • 3階層の役割分担：個人の普遍ルール／現場の前提／案件の固有ルール
  • 書くのは指示だけ。進捗・状態は書かない（陳腐化の最大要因）
  • 具体的に・「するな」より「こうして」・200行を超えたら分離
03

フォルダ構造の最適化

Claudeはフォルダ名とファイル名で世界を理解します。地図が正確なら、探索は速く、事故は減る。

AIは「名前」で理解する

Claudeが作業を始めるとき、最初にやるのはフォルダとファイルの一覧を眺めることです。つまりフォルダ名・ファイル名が、そのままAIへの説明文になります。「新しいフォルダ(2)」「その他」「aaa.md」が並んだ環境では、優秀なAIでも迷子になります。これは人間の新人と完全に同じです。

基本形は3つだけ

基本 1

1テーマ＝1フォルダ

事業・プロジェクトごとに独立したフォルダを切り、混ぜない。境界が明確だと、CLAUDE.mdもナレッジもそのフォルダに紐づけられます。

基本 2

番号つきサブフォルダ

01_契約/ 02_顧客管理/ のように番号＋役割名で並べると、構造が一覧で伝わり、Claudeへの「ここに何があるか」の説明が不要になります。

基本 3

生成物はoutput/に分離

AIが作るファイルの置き場を1つに決める。資料の原本と生成物が混ざらず、「消していいもの」の判断も安全になります。

ビフォー / アフター

Before：AIが迷う構造
デスクトップ/
├── 新しいフォルダ/
├── 新しいフォルダ(2)/
├── 請求書_最新_これ使う.xlsx
├── 請求書_最新2.xlsx
├── メモ.txt
├── A社資料/（B社の資料も入っている）
└── いろいろ/

どれが正か分からない重複、名前と中身の不一致。Claudeは推測で動くしかなく、推測は事故のもと。

After：AIが迷わない構造
仕事/
├── A社プロジェクト/
│   ├── CLAUDE.md   ← この現場のルール
│   ├── 01_契約/
│   ├── 02_資料/
│   ├── 03_議事録/
│   └── output/     ← 生成物はここ
└── B社プロジェクト/
    ├── CLAUDE.md
    └── …

名前を読むだけで構造が分かる。正本の場所が一意。Claudeへの説明コストがほぼゼロになる。

「どこで起動するか」も設計のうち

Claude Codeは起動したフォルダを仕事場とみなし、そこのCLAUDE.mdを読み込みます。つまり起動場所の選択は「今日はどの職場に出社してもらうか」の選択です。

  • A社の作業だけする日 → A社プロジェクト/ で起動。A社のルールと文脈だけが読み込まれ、集中力が上がる
  • 複数プロジェクトを横断する作業 → 親フォルダ（仕事/）で起動。全体を見渡せる代わりに、文脈は薄くなる

「深く潜って起動すれば専門家、浅く起動すれば横断役」。この使い分けを覚えると、同じ環境でも精度が変わります。

この章の要点
  • フォルダ名・ファイル名はAIへの説明文。名前で分かる構造にする
  • 基本形：1テーマ1フォルダ／番号つきサブフォルダ／output/分離
  • 起動場所＝出社先。専門作業は深く、横断作業は浅く起動する
04

知識をどこに置くか — 記憶の設計

「全部CLAUDE.mdに書く」が破綻する理由と、知識を4つに書き分ける考え方。

使い込むほど、残したい知識は増えていきます。ルール、決まった仕様、顧客の事情、今の進捗、あなたの好み。これを全部CLAUDE.mdに入れると、肥大化と陳腐化で崩壊します。知識は寿命と用途が違うものを、別の場所に置くのが原則です。

4つの置き場所

置き場所入れるもの寿命例
CLAUDE.md毎回守らせたい「指示・ルール」ほぼ永続「表はGoogle Sheetsで」「削除前は確認」
ナレッジファイル
（例：KNOWLEDGE.md）		確定した「事実」の台帳長い（訂正あり）用語の定義、決定事項、仕様、注意点
進捗メモ
（例：00_進捗.md）		「いまの状態」と次の一手短い（毎回更新）「デザイン修正まで完了。次はフォーム実装」
AIの自動メモリあなたの「好み」・横断的な行動修正長い「絵文字は使わない」「推奨案で進めてほしい」

覚え方は「ルールはCLAUDE.md、事実は台帳、状態は進捗メモ、好みはメモリ」。どこに書くか3秒で判断できるようになれば、環境は散らかりません。

陳腐化と戦う3原則

環境設計の最大の敵は、古い情報です。古いルールや状態が残っていると、Claudeはそれを「現在の真実」として扱ってしまいます。

  • 状態は一箇所にだけ書く — 同じプロジェクトの進捗を複数のファイルに書くと、必ず食い違いが生まれます。正本を1つ決める
  • 変わったら追記ではなく上書き — 「※6/20訂正：上記は旧仕様」と追記する方式は、古い記述が残り続けます。行そのものを書き換える
  • CLAUDE.mdに状態を書かない — 第2章の原則1。毎回読み込まれる場所に置いた古い状態は、最も害が大きい

環境を育てる「失敗→ルール化」ループ

知識の置き場が決まると、環境を育てる型ができます。

失敗が起きる期待と違う動き・誤解・事故
→
原因を言語化何を知らなかったから起きた？
→
1行ルール化CLAUDE.md か台帳に追記
→
同じ失敗が消える次のセッションから自動で効く
AIを叱っても次のセッションには忘れている。ファイルを直せば、二度と忘れない。

このループを回している人と回していない人の環境は、1か月で別物になります。Claude Codeが「使うたびに賢くなる」ように感じられたら、環境設計が機能している証拠です。

この章の要点
  • 知識は4つに書き分ける：ルール／事実／状態／好み
  • 陳腐化対策：状態は一箇所・追記でなく上書き・CLAUDE.mdに状態を書かない
  • 失敗したらAIを叱らずファイルを直す。これが環境を育てる唯一のループ
05

定型業務の部品化 — コマンドとスキル

「毎回うまく頼めるか」に賭けるのをやめて、うまい頼み方をファイルに封じ込める。

サインは「同じ説明を3回」

週報の作成、議事録の整形、リサーチのまとめ方——同じ趣旨の指示を3回出したら、それは部品化のサインです。.claude/commands/ フォルダに手順を書いたMarkdownファイルを置くだけで、自作のスラッシュコマンドになります。

.claude/commands/weekly-report.md　→　「/weekly-report」で呼べる
今週の週報を作成してください。

手順:
1. 03_議事録/ の今週分を読む
2. 「完了・進行中・課題」の3部構成で要約する
3. output/ に「YYYY-MM-DD_週報.md」として保存する

文体: 敬体。全体でA4半分以内。

ポイントは、これが単なる時短ではないことです。部品化には3つの効果があります。

効果 1

品質が揺れない

頼み方の出来不出来に左右されず、毎回同じ水準の結果になる。「今日はうまく伝わらなかった」が消えます。

効果 2

改善が蓄積する

結果が気に入らなければコマンドのファイルを直す。直した分は次回から必ず効く。頼み方が資産になります。

効果 3

人に渡せる

プロジェクトのフォルダに置けばチームの共有財産に。あなたの「うまい頼み方」を他の人もそのまま使えます。

スキルとの関係：コマンドをさらに発展させて、手順書＋参考資料＋テンプレートを1つのフォルダにまとめたものが「スキル」です。まずはコマンド化から始めて、付属資料が必要になったらスキルに育てる、という順番で十分です。
この章の要点
  • 同じ説明を3回したら部品化のサイン
  • コマンド化は時短ではなく品質の安定装置。頼み方が資産になる
  • コマンド → 資料つきのスキルへ、必要になってから育てる
06

権限とHooksで「守らせる」

CLAUDE.mdは「お願い」。守られる確率は高いが、100%ではない。絶対に守らせたいことは仕組みにする。

権限設計＝決裁権限規程

Claude Codeは操作ごとに許可を求めてきますが、毎回確認に答えるのは現実的ではありません。そこでallow（自動で実行してよい）／ask（毎回確認）／deny（禁止）の3段階を事前に設計します。新入社員の決裁権限を決めるのと同じ発想です。

段階割り当てるもの考え方
allow読み取り・検索・output/への保存など失敗してもやり直せる操作は自動化してテンポを守る
ask外部への送信・機密ファイルへのアクセス影響が外に出る操作は、人間の目を1回はさむ
denyメール送信・強制削除・認証情報の読み取り取り返しがつかない操作は、そもそも実行不能にする

設計の軸は「危険なもの以外は自動承認、取り返しのつかないものだけ確認か禁止」。全部確認にすると遅くて使わなくなり、全部自動にすると事故が起きます。この中間を意図して設計します。

お願い（CLAUDE.md）と強制（Hooks）の使い分け

CLAUDE.mdに「削除の前は確認して」と書けば、ほぼ守られます。しかしAIの判断を経由する以上、100%の保証はありません。そこでHooks（フック）——決まったタイミングで必ず実行される自動処理——の出番です。

CLAUDE.mdに書く（お願い）

「ファイルを消すときはゴミ箱経由にしてください」

ほとんどの場面で守られる。ただしAIが文脈を誤解すれば、すり抜ける可能性はゼロではない。

Hooksで仕組みにする（強制）

削除コマンドを検知したら、実行直前に自動でゴミ箱行きコマンドへ置き換える処理を仕込む

AIの判断を経由しないため、例外なく実行される。「うっかり」が構造的に起きない。

使い分けの基準は損害の大きさです。破られても困らないルールはCLAUDE.mdへ、破られたら取り返しがつかないルールはHooksと権限（deny）へ。この二段構えができると、安心して大きな仕事を任せられるようになります。

この章の要点
  • 権限は決裁権限規程：allow／ask／denyを事前に設計する
  • 軸は「危険なもの以外は自動、取り返しのつかないものだけ確認・禁止」
  • CLAUDE.md＝お願い、Hooks＝強制。損害の大きさで使い分ける
07

セッション運用術

環境が整っても、セッションの使い方が雑だと精度は落ちます。日々の運用の3つの規律。

規律1：1仕事1セッション

Claudeのコンテキスト（作業机）には、会話のすべてが載っています。A社の仕事が終わった机の上でB社の仕事を始めると、無関係な文脈が判断を濁らせます。仕事の区切りで /clear（会話のリセット）。これだけで精度のブレがかなり消えます。

規律2：セッションをまたぐ仕事は「引き継ぎメモ」を書かせる

大きな仕事は1セッションで終わりません。そして次のセッションのClaudeは、今日の作業を何も覚えていません。終える前に「進捗と次の一手を進捗メモに書いて」と頼む習慣をつけると、翌日のClaudeがメモを読んでそのまま続きから走れます。第4章の「進捗メモ」はこのためにあります。

規律3：大きな作業はプランモードから

いきなり実行させると、方向違いの作業が数十分進んでしまうことがあります。プランモード（読み取り専用で計画だけ立てるモード）でまず計画を出させ、合意してから実行。手戻りが減り、意図しない変更も起きません。「大工事の前に見積もりと図面を確認する」のと同じです。

コンテキストが苦しくなったら：長い作業の途中で反応が鈍く感じたら、/compact（要約して圧縮）で机を片付けられます。ただし本質的な対策は、仕事を区切って規律1・2を守ることです。
この章の要点
  • 1仕事1セッション。区切りで /clear
  • またぐ仕事は引き継ぎメモを書かせてから終える
  • 大きな作業はプランモードで合意してから実行
